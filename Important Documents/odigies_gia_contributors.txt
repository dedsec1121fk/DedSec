DEDSEC PROJECT DOCUMENTATION (ΕΛΛΗΝΙΚΗ ΕΚΔΟΣΗ)

1. Γενική Φιλοσοφία
Το DedSec Project δημιουργήθηκε με έμφαση στην πρακτική υλοποίηση, τη δημιουργικότητα και τη συνεχή εξέλιξη. Παρότι αρκετά μέρη του project ξεκίνησαν με πιο γρήγορη και πειραματική λογική ανάπτυξης, ο στόχος παραμένει σταθερός: κάθε εργαλείο και κάθε αρχείο να λειτουργεί σωστά, να έχει καθαρή δομή και να προσφέρει πραγματική χρησιμότητα.

2. Κατεύθυνση Ανάπτυξης
Η ανάπτυξη του project πρέπει να βασίζεται σε εργαλεία και μεθόδους που βοηθούν στη δημιουργία σωστών, λειτουργικών και νόμιμων scripts. Για νέα scripts, διορθώσεις ή βελτιώσεις, μπορούν να χρησιμοποιούνται AI εργαλεία όπως το ChatGPT, αρκεί τα prompts να είναι ξεκάθαρα, να υπάρχει συνέχεια στο ίδιο chat όταν χρειάζεται debugging, και κάθε αποτέλεσμα να ελέγχεται πριν προστεθεί στο project.
Για απλά scripts, χρήσιμη πρακτική είναι:
- ξεκάθαρη περιγραφή του στόχου,
- αποστολή του προβλήματος με σαφήνεια,
- συνέχιση στο ίδιο chat για fixes και βελτιώσεις,
- απαίτηση να επιστρέφεται το αρχείο έτοιμο για λήψη, όταν αυτό είναι απαραίτητο.

Για πιο απαιτητική ανάπτυξη, debugging και καλύτερα αποτελέσματα, είναι προτιμότερο να χρησιμοποιείται κάποιο paid plan του ChatGPT που δίνει πρόσβαση σε πιο ισχυρές δυνατότητες. Αυτό βοηθά ιδιαίτερα όταν χρειάζονται πιο σύνθετες απαντήσεις, καλύτερη ανάλυση προβλημάτων και πιο ολοκληρωμένα παραδοτέα αρχεία.

Άλλα AI εργαλεία μπορούν επίσης να χρησιμοποιηθούν υποστηρικτικά. Το Gemini μπορεί να είναι χρήσιμο για πιο απλές εργασίες, αλλά κάθε αποτέλεσμα πρέπει να ελέγχεται προσεκτικά για λάθη. Το DeepSeek μπορεί επίσης να χρησιμοποιηθεί ως βοηθητικό εργαλείο για ιδέες, debugging και draft code, όμως κάθε παραγόμενο αρχείο πρέπει πάντα να ελέγχεται για ποιότητα, σταθερότητα, ασφάλεια, συμβατότητα και νόμιμη χρήση πριν ενσωματωθεί στο DedSec Project. Αυτό ισχύει ακόμη περισσότερο σε περιπτώσεις όπου το output ενός AI εργαλείου ακουμπά ευαίσθητες, ριψοκίνδυνες ή οριακές τεχνικά περιοχές.

3. Ποιότητα και Παρουσίαση
Το DedSec Project πρέπει να διατηρεί ομοιόμορφη αισθητική και καθαρή παρουσίαση. Ό,τι περιέχει το project οφείλει να δείχνει προσεγμένο, όμορφο, λειτουργικό και όχι πρόχειρο ή “ψεύτικο”.
Κάθε script, εργαλείο ή αρχείο πρέπει:
- να είναι σωστά οργανωμένο,
- να ανήκει στην κατηγορία που του ταιριάζει,
- να τοποθετείται με βάση τον πραγματικό του σκοπό και τη χρήση του,
- να συμβάλλει ουσιαστικά στη συνοχή του project.

4. Οργάνωση Αρχείων και Κατηγοριών
Κάθε νέο script πρέπει να καταχωρίζεται στην κατάλληλη κατηγορία ανάλογα με τη λειτουργία του. Η κατηγοριοποίηση δεν γίνεται τυχαία ή μόνο αισθητικά, αλλά με βάση το τι κάνει το script, σε τι χρησιμεύει και πού ανήκει λογικά μέσα στο project.
Εάν στο μέλλον προστεθούν scripts που απαιτούν root, τότε πρέπει να δημιουργηθούν ξεχωριστοί φάκελοι, σαφής διαχωρισμός δομής και αντίστοιχες ενημερώσεις σε README, documentation και ιστοσελίδα, ώστε να είναι ξεκάθαρο ποια εργαλεία είναι για απλή χρήση και ποια απαιτούν root πρόσβαση.

5. Δομή Scripts και Παραγόμενων Αρχείων
Κάθε script στο DedSec Project πρέπει κατά προτίμηση να παραμένει όσο πιο μικρό και καθαρό γίνεται σε επίπεδο δομής. Ο βασικός κανόνας είναι ότι κάθε script δεν πρέπει να απλώνεται άσκοπα σε πολλά αρχεία, ώστε το tree του project να παραμένει minimal, καθαρό και εύκολα διαχειρίσιμο.
Όπου είναι πρακτικά δυνατό, κάθε script πρέπει να παραμένει σε ένα κύριο αρχείο. Αν χρειάζεται να παράγει αρχεία, outputs, exports ή αποθηκευμένα δεδομένα, είναι προτιμότερο αυτά να τοποθετούνται μέσα σε ξεχωριστό φάκελο με το όνομα του ίδιου του script, ώστε:
- να αποφεύγεται η ακαταστασία,
- να είναι πιο εύκολη η συντήρηση,
- να μην μπερδεύονται generated files με βασικά αρχεία του project,
- να διατηρείται καθαρή η συνολική δομή του repository.

6. Δίγλωσση Συντήρηση
Οι ελληνικές και οι αγγλικές εκδόσεις του project πρέπει να παραμένουν πάντα συγχρονισμένες και ενημερωμένες. Οποιαδήποτε νέα προσθήκη, διόρθωση ή αλλαγή πρέπει να περνά και στις δύο γλώσσες.
Εάν υπάρξει:
- λάθος μετάφραση,
- σπασμένο αρχείο,
- ασυνέπεια μεταξύ ελληνικής και αγγλικής έκδοσης,
τότε η διόρθωση πρέπει να γίνεται όσο το δυνατόν γρηγορότερα.

7. Κανόνες για την Ιστοσελίδα
Η ιστοσελίδα του project δεν πρέπει να αλλάζει με ακραίο τρόπο σε τακτική βάση. Πρέπει να υπάρχει συνοχή στο design, στη διάταξη και στη συνολική εμπειρία χρήσης.
Η βασική προτεραιότητα του project είναι τα tools. Νέες σελίδες, βελτιώσεις ή επεκτάσεις στην ιστοσελίδα πρέπει να γίνονται με προσοχή, χωρίς να αλλοιώνεται ο χαρακτήρας της.
Για μεγάλες αλλαγές ή προσθήκες στην ιστοσελίδα, προτείνεται να δίνεται ολόκληρη η υπάρχουσα ιστοσελίδα στο εργαλείο ανάπτυξης ή στο AI που χρησιμοποιείται, μαζί με ξεκάθαρη περιγραφή του ζητούμενου, και να επιστρέφεται το αποτέλεσμα ως πλήρες πακέτο αρχείων για άμεσο έλεγχο.

8. Αντίγραφα Ασφαλείας
Πρέπει να υπάρχουν πάντα backup των τελευταίων σωστών και σταθερών εκδόσεων όλων των βασικών στοιχείων του project:
- scripts,
- ιστοσελίδα,
- μεταφράσεις,
- documentation,
- readme αρχεία,
- βοηθητικά assets.
Ο στόχος είναι να μην “σπάσει” ποτέ ούτε το project ούτε η ιστοσελίδα χωρίς δυνατότητα άμεσης επαναφοράς.

9. Προσβασιμότητα και Κοινό
Το project πρέπει να είναι εύκολα κατανοητό και προσβάσιμο ακόμη και σε αρχάριους χρήστες, χωρίς όμως να αποκλείει πιο προχωρημένα εργαλεία και δυνατότητες.
Με απλά λόγια:
- πρέπει να μπορεί να το χρησιμοποιήσει και να το καταλάβει ένας λιγότερο έμπειρος χρήστης,
- αλλά είναι αποδεκτό να περιέχει και advanced λειτουργίες, αρκεί να είναι σωστά οργανωμένες και επεξηγημένες.

10. Επιλογή Νέων Ιδεών και Εργαλείων
Οποιαδήποτε ιδέα βρεθεί σε πλατφόρμες όπως YouTube, Instagram ή TikTok μπορεί να αξιολογηθεί για ένταξη στο project, εφόσον είναι πραγματικά χρήσιμη, εφαρμόσιμη και έχει ουσιαστική αξία.
Δεν πρέπει να προστίθεται κάτι μόνο και μόνο επειδή φαίνεται εντυπωσιακό. Το βασικό κριτήριο είναι αν όντως προσφέρει κάτι χρήσιμο στον χρήστη ή στην κοινότητα.

11. Κανόνας Χρησιμότητας
Δεν πρέπει να μπαίνει στο project κανένα script που υπάρχει μόνο για επίδειξη ή εντυπωσιασμό χωρίς πραγματική λειτουργία.
Οτιδήποτε προστίθεται πρέπει:
- να λειτουργεί αξιόπιστα,
- να έχει πραγματική χρήση,
- να προσφέρει βοήθεια ή αξία,
- να είναι κατάλληλο για υπεύθυνη και νόμιμη χρήση.
Η ποιότητα και η πραγματική χρησιμότητα έχουν πάντα προτεραιότητα απέναντι στο “φαίνεσθαι”.

12. Pull Requests, Contributors και Ομάδα
Δεν πρέπει να γίνονται δεκτά όλα τα pull requests. Κάθε πρόταση αλλαγής πρέπει να ελέγχεται προσεκτικά ως προς την ποιότητα, τη χρησιμότητα, τη συμβατότητα με το project και τη συνολική κατεύθυνση του DedSec Project.
Το να κάνει κάποιος ένα pull request δεν σημαίνει αυτόματα ότι είναι έτοιμος να θεωρηθεί contributor ή να ενταχθεί στην ομάδα συντήρησης.

Για να γίνει κάποιος contributor και να μπει στην ομάδα του DedSec Project, πρέπει να έχει περάσει σημαντικό χρονικό διάστημα μέσα στο οποίο θα έχει φανεί σταθερά:
- ο χαρακτήρας του,
- η ευγένειά του,
- ο τρόπος συνεργασίας του,
- ο σεβασμός του προς το project και την κοινότητα,
- η συνέπειά του στην ποιότητα της δουλειάς του.

Το DedSec Project δεν είναι μόνο για όσους είναι προγραμματιστές. Είναι για όσους θέλουν πραγματικά να συμβάλουν σε μια καλή αλλαγή στον κόσμο μέσα από χρήσιμα εργαλεία, σωστή νοοτροπία και υπεύθυνη στάση.
Για αυτόν τον λόγο, έχει τεράστια σημασία ποιοι φαίνονται δημόσια ως maintainers, contributors ή μέλη της ομάδας. Τα άτομα που συνδέονται με το project πρέπει να αντιπροσωπεύουν όχι μόνο τεχνική ικανότητα, αλλά και ήθος, σοβαρότητα και σωστή συμπεριφορά.

13. Τελική Αρχή
Το DedSec Project πρέπει να εξελίσσεται με συνέπεια, σωστή δομή, καθαρές κατηγορίες, ενημερωμένες μεταφράσεις, σταθερά backups και ουσιαστικά εργαλεία. Η ανάπτυξή του πρέπει να βασίζεται στη χρησιμότητα, στη σαφήνεια, στην αισθητική συνοχή και στη διατήρηση μιας αξιόπιστης εμπειρίας για όποιον το χρησιμοποιεί ή το συντηρεί.
